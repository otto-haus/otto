## Summary

Removes a direct north-star honesty break: `routine.run.manual` previously recorded a
**success receipt** referencing only `stepCount`, while `RoutineStore.runManual()` never
invoked `PracticeRunner` for the practices declared in `routine.yaml`. Routines like
`morning` and `weekly-culture` reported success without running any steps.

### Change

- General routines now **execute each declared step** via `executeSteps()`:
  - Runtime-wired practices (`charter` / `review` / `field-note`) run via shared
    `PracticeRunner`; each emits its own receipt, referenced as step evidence on the routine
    receipt.
  - Steps whose practice is **not** runtime-wired (`decision` / `follow-up`) or that fail
    are recorded honestly as `skipped` / `blocked`.
- Routine receipt is `success` **only when every step ran**; otherwise `blocked` with a
  recoverable `routine_steps_incomplete` blocker naming the steps that could not run.
- The delegated `ai-frontier-review` and `practice-mining` executors keep their real domain
  logic unchanged.
- IPC shares its `PracticeRunner` with `RoutineStore`; tests inject a hermetic runner so
  routine runs never touch `~/.otto`.
- `RoutineStore` constructor gains optional `practiceRunner` param (defaults to
  `new PracticeRunner()`); `ipc.ts` now passes the shared `practiceRunner` instance.

## Risks

- **`RoutineStore` constructor change:** New optional `practiceRunner` param added at
  position 4. The default `new PracticeRunner()` preserves backward compatibility for call
  sites that omit it. `ipc.ts` is the only production construction site and has been updated
  to pass the shared `practiceRunner`.
- **Receipt structure change:** Routine receipts for general routines now include
  `stepsRun`, `stepsSkipped`, `stepsBlocked` instead of just `stepCount`. Any consumer
  parsing only `stepCount` will see it missing; this is intentional (the field was
  misleadingly populated before).
- **`morning` routine now returns `blocked` (honest):** `morning` declares `decision` +
  `follow-up` steps that are not runtime-wired; the test confirms this is now `blocked`
  with `stepsSkipped ≥ 1`. UI must handle `blocked` routine results gracefully (no
  regression introduced — status was previously always `success` regardless).
- **`executeSteps()` running real `PracticeRunner`:** For wired steps, this involves
  real file I/O in the receipts/runs directories. Tests use hermetic paths under tmp.
